非エンジニアがAIコーディングやるにあたって、必要なツールとアプローチをまとめてみました。

# 1. VS Codeとは？
プログラミングを行うための **IDE（統合開発環境）**  
メモ帳と違い、以下が自動でできる。

✅ コードの色分け  
✅ 自動整形  
✅ 括弧の対応チェック  
✅ 補完候補の表示  
✅ GitHub と連携  
✅ AI コーディングが使える（Copilot / ChatGPTプラグインなど）

世界で最も利用されているIDEで、拡張機能が非常に多く、  
**「プログラミングができるメモ帳」ではなく  
「AIと一緒に作業するためのワークベンチ」** と考えるとわかりやすい。

---

# 2. GitHubとは？
- プログラムやドキュメントを保管するクラウドサービス
- バージョン管理でき、いつ変更したか、誰が変更したかを追跡できる
- チーム開発で必須
- 有用なソフトが無料でgithubベースで公開されていて、OSSの巨大プラットフォームになっている

| 用語 | 意味 |
|------|------|
| Git | 変更履歴を管理する仕組み（ローカルでも使える） |
| GitHub | Gitをクラウドで使えるサービス |
| Repository | プロジェクトのフォルダ |
| Commit | 変更履歴を記録すること |

AI コーディングと相性がよく、**生成AI がコードを解析しやすくなるメリット**がある。

---

# 3. Python & PowerShell  
生成 AI と相性のよい言語

| 言語 | 特徴 |
|------|------|
| Python | 機械学習・データ処理・Web など用途が広いが、インストール必要・開発環境メンテが面倒 (下記仮想環境でカバー) |
| PowerShell | Windowsに標準搭載、ツールの自動化やファイル処理が得意だが、Pythonほど複雑な処理は不可|

Python が AI に向いている理由  
✅ コンパイル不要  
✅ 世界で最も学習教材が多い  
✅ Web情報が豊富でFoundational Model (生成AI) の回答精度が高い

PowerShell が便利な理由  
✅ Windowsなら最初から使える  
✅ バッチより直感的  
✅ 自動化に強い

# 4. 仮想環境とは？（UV を使う理由）
Python は複数のプロジェクトで「違うバージョンのライブラリ」が必要になりがち。

例）  
- Aプロジェクト → numpy 1.20  
- Bプロジェクト → numpy 1.23

そのまま入れると上書き衝突して壊れる。  
⇒ **仮想環境（専用の小さなPython環境）を作ることで解決**
クローズドな開発環境を都度用意し、プロジェクトごとの衝突を避けている。

従来よく使われるツール  
- venv（標準）  
- conda（重い）  
- pipenv（遅い）

**UV は高速・軽量で、最近一気に普及**  
- 仮想環境作成  
- パッケージ管理  
- pytest 等も高速実行  
- Rust製で爆速

# 5. V字工程に従ってAIコーディング設計する
ソフトウェア開発やシステム開発で使われるプロセスモデル。  
企画 → 設計 → 実装 → テスト と進み、  
「実装に対応するテストが必ずある」のが特徴。
AI時代における

---

## ✅ V字工程の概要
     要求定義        ←→      受け入れテスト
         \                     /
          基本設計      ←→   システムテスト
             \             /
              詳細設計 ←→ 結合テスト
                 \     /
                 実　装

- 左側 = 設計（何を作るのか決める）
- 右側 = テスト（作ったものが正しいか確認）
- 実装（コードを書く）は谷の底

つまり、**「作る前に決める → 作った後に検証する」**という考え方。

---

# 6. AIコーディングでも V字工程が重要
AI にコードを書かせる時でも、V字工程に従ってやったほうが良い

| フェーズ | AI コーディングで必要なこと |なぜか|
|----------|------------------------------|
| 要求定義 | 何を実現したいかを文章で明確にする（プロンプト）。AIと対話しながら深めるもOK | AIの大局観をフルに活用する方がよい |
| 基本設計 | AIを活用しておすすめの方法を吐かせる。10個ほどアプローチを出させて一番良い設計を抽出もOK。どの方法で行くかは人間が決める。| Pythonをどうやって組むか(ライブラリの存在)は設計者は知らないケースが絶対出てくる。AIにアシストさせるほうが良い |
| 詳細設計 | 関数名、変数、データ形式、例外処理を決める。AIにアイディア出させる。| 詳細設計は今のところAIは十二分に満足な結果を出さないことが多いが素案は出させる。今のところ例外処理多めになる。 |
| 実装 | AIにコードを書かせる。定期的にリファクタリングの指示をしてコードの設計見直し・品質向上を行う。[SOLIDの原則と実装](https://qiita.com/baby-degu/items/d058a62f145235a0f007) | 実装めちゃめちゃ早いが、ファイル数がどんどん増えてしまう。収束と発散を繰り返させてまとめる。 |
| 単体テスト | 小さな処理単位で動作確認（AIにテストコードを書かせても良い。Unit testしてで伝わる） | やってくれるのでお願いしちゃう |
| 結合テスト | 部品を組み合わせた時の動作確認 (結合テストしてで伝わる) |やってくれるのでお願いしちゃう |
| システムテスト | 想定通りに動くか、異常系も含め確認 (システムテストしてで伝わる) |やってくれるのでお願いしちゃう |

---

## ✅ ありがちな失敗例
| 失敗 | なぜ起こる？ |
|------|--------------|
| いきなりコードを書かせる | 良い設計になってないままコード書くと修正工数激増。AIにより良いアプローチ/設計を提案させながらやる (V字工程の上流からやる) |
| 設計で時間かけすぎ | コーディングで時間のかからない時代、とりあえず沢山のアプローチでコーディングさせてうまくいくものを抽出というのも一つの考え方 |
| コードは動くのに使えない | テスト視点がない |
| 同じバグを何度も修正 | 小さい単位のテストをしていない |
| 定期的にリファクタリング | SOLIDの原則参照 |

つまり、  
👉 **正しい要件、正しい設計/アプローチでないと結局遠回り**  
👉 **V字工程で設計の段階からAIを活用**
👉 **極論10個アプローチ提案させてAIのおすすめ、設計者の考えを勘案して上位3つを全部コーディングさせてうまくいくものを採用というのもあり**

# 7. まとめ
| 項目 | メリット |
|------|----------|
| VS Code | AIと組み合わせて最強の作業環境 |
| GitHub | コードを保存・共有・履歴管理できる |
| Python & PowerShell | AIがアシストしやすく初心者向き |
| UVの仮想環境 | 依存関係が壊れず、セットアップが楽 |
| V字工程 | 設計段階からAIに正しい方向へ導かせる。品質ばらつき低減、工数低減 |

AIコーディングは  
**「自分で全部書く時代」 → 「AIに設計からやらせて、書かせて、テストする時代」**

人力だと**書く**プロセスがボトルネックだが、AI時代それシフトしつつある。AIに流し込むアイディアの物量がボトルネックになる。パラレルに各プロセスを稼働させ、成功確率を上げることが重要。
